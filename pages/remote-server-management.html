Changing the Root Password

The first and most fundamental server security task

When you're already logged in as root, changing your password is a single command. No old password required.

bash
passwd

# Output:
Changing password for user root.
New password:
Retype new password:
passwd: all authentication tokens updated successfully.
Tip: Password input is invisible — no characters appear as you type. This is intentional and normal on all Linux systems.
Best practice: Use a strong password (12+ chars, mixed case, numbers, symbols) and store it in a password manager like 1Password or Bitwarden. Never reuse passwords across servers.

Creating a New User

Important: behaviour differs between distros

On Debian/Ubuntu, adduser is interactive and prompts for a password. On CentOS/RHEL/AlmaLinux, it is just an alias for useradd — silent, no prompt.

Debian / Ubuntu

adduser adminuser — interactive wizard, sets password, creates home directory automatically.

CentOS / RHEL

adduser adminuser — silent. Creates user and home directory but does not set a password. You must run passwd adminuser separately.

bash — CentOS/RHEL
# Step 1: create the user
adduser adminuser

# Step 2: set their password (required on RHEL-based systems)
passwd adminuser

# Step 3: verify the user exists
id adminuser
uid=1002(adminuser) gid=1002(adminuser) groups=1002(adminuser)
How it works: Linux stores user accounts in /etc/passwd, passwords (hashed) in /etc/shadow, and group memberships in /etc/group. You can inspect these files directly if needed.

Granting Sudo Access

The sudo group is named differently depending on the distro

The sudo group name is a common source of confusion when switching between Linux distributions.

Distribution Sudo Group Name Command
Debian / Ubuntu sudo usermod -aG sudo adminuser
CentOS / RHEL / AlmaLinux wheel usermod -aG wheel adminuser
bash
# Add adminuser to the wheel group (CentOS/RHEL)
usermod -aG wheel adminuser

# Verify group membership
groups adminuser
adminuser : adminuser wheel   # ✓ wheel confirms sudo access
Flag explained: In usermod -aG, the -a means append (don't remove existing groups) and -G specifies supplementary groups. Omitting -a would replace all existing groups — a common mistake.

Switching Between Users

Navigating user contexts in an active terminal session

bash
# Switch to adminuser with a full login shell
su - adminuser

# Confirm you're now adminuser
whoami
adminuser

# Test sudo works
sudo whoami
root

# Return to previous user
exit
The - flag matters: su adminuser switches user but keeps your current environment. su - adminuser (with dash) creates a full login shell — loading adminuser's ~/.bashrc, PATH, and environment variables. Always prefer su -.

SSH Login from Terminal

Connecting to a remote server securely

1

Connect to the server

bash — run on your Mac
ssh adminuser@vps01.example.net
# or using IP address
ssh adminuser@203.0.113.10
2

Enter your password when prompted

Once authenticated, you'll see your server's shell prompt: [adminuser@vps01 ~]$

Pro tip — SSH Keys: Instead of typing a password every time, set up SSH key authentication. Run ssh-keygen on your Mac, then ssh-copy-id adminuser@vps01.example.net. Future logins become passwordless and more secure.

Verifying Host Fingerprints

Protecting against man-in-the-middle attacks

On first connection to any server, SSH presents a fingerprint for you to verify. This is a critical security step — never blindly accept it.

bash — run ON the server (as root)
# Get the server's actual ED25519 fingerprint
ssh-keygen -lf /etc/ssh/ssh_host_ed25519_key.pub

256 SHA256:lLM3w0IddUPLOWeCxYG8RFH0qMRxW5C7L/tz5z+hAkQ no comment (ED25519)

# This must exactly match what SSH shows during connection
ScenarioWhat it meansAction
Fingerprints match Server is genuine Type yes to continue
Fingerprints differ Possible MITM attack or server rebuild Do NOT connect — investigate immediately
After a server rebuild: If your server was reinstalled, it gets new SSH host keys. SSH will warn you with "REMOTE HOST IDENTIFICATION HAS CHANGED". Remove the old key with ssh-keygen -R vps01.example.net then reconnect and reverify.

Reading Server Logs

The essential skill for understanding what your server is doing

1

System messages log

bash
sudo tail -f /var/log/messages          # live follow
sudo tail -100 /var/log/messages        # last 100 lines
2

Apache error log (cPanel path)

bash
sudo tail -100 /etc/apache2/logs/error_log   # cPanel
sudo tail -100 /etc/httpd/logs/error_log      # standard RHEL
3

Systemd journal (most powerful)

bash
# Errors only, within a date range
sudo journalctl --since "2026-03-01" --until "2026-03-15" -p err

# Follow live errors
sudo journalctl -f -p err

# Errors from a specific service
sudo journalctl -u httpd -n 50
4

Search logs for keywords

bash
sudo grep -i "error\|panic\|oom\|kill" /var/log/messages | tail -50
Press q to exit when using journalctl in pager mode. Press Ctrl+C to stop live log following with tail -f.

Understanding Firewall Logs

Decoding the noise in /var/log/messages

Firewall block logs are normal background noise on any public-facing server. Here's how to read them:

log sample
Mar 15 07:27:46 server1 kernel: Firewall: *UDP_IN Blocked*
  IN=eth0  SRC=199.192.19.170  DST=255.255.255.255
  PROTO=UDP  SPT=11002  DPT=11002

Mar 15 07:27:57 server1 kernel: Firewall: *TCP_IN Blocked*
  IN=eth0  SRC=167.94.146.32  DST=203.0.113.10
  PROTO=TCP  DPT=9235  SYN  URGP=0
Log TypeKey IndicatorsMeaningThreat
UDP Broadcast Blocked DST=255.255.255.255 Network discovery broadcasts — not targeting your server None
TCP Port Scan Blocked SYN + random DPT Bot scanning for open ports — internet background noise Low
SSH Brute Force PAM-hulk: TOO MANY FAILURES Automated login attempts on SSH — being blocked by cPHulk Medium
OOM Kill Out of memory: Killed process Server ran out of RAM — kernel killed MySQL to survive Critical
Your firewall is working: Seeing lots of "Blocked" entries is good — it means the firewall is actively rejecting unwanted traffic. Concern arises when you see allowed connections to unexpected ports.

Investigating Server Downtime

A systematic diagnostic checklist

When your sites go down and you have to reboot, use this checklist to find the cause.

1

Check reboot history

bash
last reboot | head -20
2

Check for OOM kills

bash
sudo grep -i "out of memory\|oom\|killed process" /var/log/messages
3

Check RAM and swap

bash
free -h
4

Check disk space

bash
df -h     # danger zone if any partition > 85%
5

Check for failed services

bash
sudo systemctl --failed
6

Deep journal error scan

bash
sudo journalctl --since "2026-03-01" -p err --no-pager

Out of Memory (OOM) Crashes

The root cause of the server's recurring downtime

What was found on this server:
Mar 01 03:36:37 kernel: Out of memory: Killed process 488238 (mysqld)
Mar 13 21:29:08 kernel: Out of memory: Killed process 762508 (mysqld)

MySQL was killed by the Linux kernel on both crash dates because the server ran out of RAM.

The server's memory situation at time of investigation:

output of: free -h
              total     used      free    shared  buff/cache  available
Mem:          1.9Gi    1.0Gi    101Mi     9.0Mi      795Mi      717Mi
Swap:            0B       0B       0B             ← NO SWAP AT ALL

Only 101MB RAM Free

Critically low headroom. Any spike in traffic or a cron job can push it to zero.

Zero Swap Space

When RAM fills up, the kernel has no fallback — it immediately kills the largest process (MySQL).

Disk at 78% (29GB/40GB)

Monitor this. At 85–90%, some services start misbehaving. Clean up logs, backups, or old files.

What happens during an OOM event:
  1. RAM fills up completely
  2. Kernel OOM killer activates
  3. MySQL (largest process) gets killed
  4. cPanel services start failing (PHP-FPM, Dovecot, chronyd)
  5. Sites become unreachable
  6. Cascading failures — reboot required

Adding Swap Space

The most critical fix to prevent future OOM crashes

Swap is disk space used as overflow RAM. When physical RAM is exhausted, the kernel moves inactive memory pages to swap instead of killing processes. Essential for any server with limited RAM.

bash — run as root or with sudo
# 1. Create a 2GB swap file
sudo fallocate -l 2G /swapfile

# 2. Secure it — only root should read it
sudo chmod 600 /swapfile

# 3. Format it as swap space
sudo mkswap /swapfile

# 4. Enable it immediately
sudo swapon /swapfile

# 5. Make it permanent across reboots
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

# 6. Verify it's active
free -h
              total     used     free
Swap:         2.0Gi      0B     2.0Gi   ✓
How much swap? A common rule of thumb: use swap equal to your RAM size. For a 2GB RAM server, 2GB swap is appropriate. For servers with 8GB+ RAM, 4GB swap is usually sufficient. Swap is slower than RAM — it's a safety net, not a performance boost.
Long-term fix: Swap prevents crashes but doesn't solve the underlying memory pressure. Consider upgrading your VPS plan to one with more RAM, or optimizing MySQL and Apache memory settings in cPanel's WHM.

Downloading Files from a Server

Copying log files to your local machine using SCP

SCP (Secure Copy Protocol) transfers files over SSH. Always run it from your local machine — not from inside the server.

1

Save the file on the server first

bash — on the server
sudo journalctl --since "2026-03-01" -p err > /tmp/server_errors.txt
2

Download to your Mac

bash — on your Mac (new terminal tab)
scp adminuser@vps01.example.net:/tmp/server_errors.txt \
    ~/Desktop/server_errors.txt
One-liner alternative: Do it all in one command from your Mac without saving to a temp file first: 
ssh adminuser@vps01.example.net "sudo journalctl -p err" > ~/Desktop/errors.txt
CommandDirectionUse case
scp user@host:/remote/file /local/path Server → Mac Download logs, backups, files
scp /local/file user@host:/remote/path Mac → Server Upload configs, scripts, files
scp -r user@host:/remote/dir /local/dir Server → Mac (folder) Download entire directories
