The Incident — April 2, 2026

Users reporting slowness — starting the investigation

At approximately 09:42 WAT, users of the websites hosted on vps01.example.net reported the server feeling slow. This guide documents the complete diagnostic process that followed — from initial check to full root cause identification.

Server Details

vps01.example.net
IP: 203.0.113.10
CentOS/RHEL 8 — cPanel managed
4 CPU cores, 5.8Gi RAM

Hosted Sites

portal-app.example.net
regional-app.example.net
service-app.example.net
records-app.example.net
All under user: appuser

Initial Symptom

Users reporting slowness at ~09:42. Server not down — responding but sluggish. Load spike suspected.

Outcome

Root cause confirmed: cPanel cron job clean_user_php_sessions hammering a slow /tmp filesystem. No attack — fully internal.

What this guide teaches: How to diagnose a live Linux server performance incident systematically — reading process tables, interpreting load averages, analysing disk I/O, auditing cron jobs, checking network connections, and distinguishing an internal cron problem from an external attack.

First Diagnostic Round

The five commands to run immediately when slowness is reported

SSH in as root and run these in sequence. Each answers a specific question about the server's health.

1

Check uptime and load average

The load average tells you instantly whether the server is overwhelmed. Compare with nproc (number of CPU cores).

bash
uptime
 09:42:17 up 8 days, 7:15, 1 user, load average: 7.20, 5.80, 6.31

nproc
4   # 4 cores — load of 7.20 means server is 1.8× overloaded
Rule: If load average > number of CPU cores, the server is overloaded. Load 7.20 on a 4-core server = 180% over capacity. This instantly confirms performance problems are real.
2

Check RAM and swap

bash
free -h
              total    used    free   shared  buff/cache  available
Mem:          5.8Gi   4.3Gi   278Mi    9.0Mi       1.2Gi     1.2Gi
Swap:         2.0Gi   421Mi   1.6Gi

Only 278Mi free RAM — tight but not critical. 421Mi of swap in use means memory pressure existed recently. The server has been using swap as a relief valve.

3

Check disk space

bash
df -h
Filesystem   Size  Used  Avail Use%  Mounted on
/dev/vda2    118G   30G   83G  27%   /
/dev/loop0   3.9G   56K  3.7G   1%  /tmp   # loop-mounted — suspect

Main disk fine at 27%. But /tmp is mounted on a loop0 device (a file pretending to be a disk) — this will become significant later.

4

Find what's consuming resources (top processes)

bash
ps aux --sort=-%cpu | head -15
USER     PID  %CPU %MEM  STAT COMMAND
appuser  ...  25.0  0.9   S   php-cgi portal-app.example.net
appuser  ...  24.0  0.9   S   php-cgi portal-app.example.net
appuser  ...   0.5  0.9   S   php-cgi service-app.example.net
... (43 php-cgi processes total)
PHP-CGI Avalanche identified: 43 simultaneous PHP-CGI workers — 36 of them for portal-app.example.net. All in S (sleeping/waiting) state. They are not computing — they are stalled waiting on something.
5

Count PHP workers specifically

bash
ps aux | grep php-cgi | grep -v grep | wc -l
43   # normal is <10 for this server

Deep Dive — Second Diagnostic Round

vmstat, iostat, and network checks — 8 minutes after the first check

By 09:50 the load had already dropped to 0.21 — the spike was short-lived and self-resolving. The second round focused on understanding what happened and why the I/O latency was so high.

1

vmstat — memory, swap, and I/O wait

Run vmstat 1 10 to get 10 samples one second apart. Key columns to watch:

bash
vmstat 1 10
procs ----memory---- ---swap-- --io-- --cpu--

 0  0 421504 1.2Gi    0    1    18    20    1

 0  0 421504 1.2Gi    0    0     0     0    0

siSwap pages read IN from disk per secAny non-zero value
soSwap pages written OUT to disk per secAny non-zero value

At 09:50: si=0 so=0 — swap no longer active. wa=1 — I/O wait normal. Recovery confirmed.

2

iostat — the disk latency revelation

iostat -x 1 3 shows per-device I/O statistics. The r_await and w_await columns expose latency.

bash
iostat -x 1 3
Device    r/s    w/s   r_await  w_await  %util
vda      3.17   7.64    44ms      6.96     1.51
loop0    0.00   0.16   131ms     83.39     1.12   # ← /tmp device
Critical finding: loop0 (which is /tmp) has a 131ms read await and 83ms write await. A healthy SSD should be 1–5ms. This means any process reading or writing to /tmp waits ~130 times longer than expected. PHP session files are stored in /tmp — every PHP worker stalls when accessing sessions.
3

Network connections — rule out a traffic flood

bash
ss -s
TCP: 12 (estab 10, closed 0, orphaned 0, timewait 0)
# Only 10 connections — this is NOT a traffic flood
Key lesson from this round: The recovery at 09:50 is itself a clue. Something fired at 09:39, caused a spike for ~8 minutes, then stopped. This pattern is characteristic of a scheduled cron job, not a traffic attack (which would be sustained). The I/O latency on /tmp is the mechanism-of-harm. Finding what accesses /tmp on a schedule is the next step.

Attack vs. Scheduled Job — How to Determine Which

Gathering evidence to distinguish a malicious flood from an internal cron trigger

Before investigating cron jobs, it's important to rule out the possibility that the spike was caused by a malicious attacker. The evidence-based approach:

Signs of an Attack

Sustained high connection count. Many different IPs in access logs. HTTP requests flooding in. Traffic doesn't stop until firewall blocks it.

Signs of a Cron Job

Spike starts at a predictable time (e.g. :09 or :39). Spike resolves itself after a few minutes. Zero or near-zero web traffic during spike. Repeats on a schedule.

The Definitive Test

Count HTTP requests in the Apache access log during the exact spike window. Zero requests = no traffic attack. Case closed.

The log check — the most important command in this investigation:

bash — run as root
# Count requests in Apache access log during the exact spike window
grep "02/Apr/2026:09:3[89]\|02/Apr/2026:09:4[0-5]" \
  /usr/local/apache/domlogs/portal-app.example.net | wc -l

0   # ZERO HTTP requests during the entire spike window — attack ruled out
Apache log path on cPanel servers: /usr/local/apache/domlogs/sitename.com (non-SSL) and /usr/local/apache/domlogs/sitename.com-ssl_log (HTTPS). These are the definitive record of all web requests that reached the server.

Security finding noted during investigation:

SSH Brute Force — separate issue: The root login greeting showed:
"There were 396 failed login attempts since the last successful login"
Source IP: 185.208.159.109. Last successful root login was March 20 — 13 days prior. cPHulk is blocking the attempts, but the volume is high. This is an unrelated ongoing threat to address separately (see Section 12).

Auditing All Cron Jobs on a cPanel Server

Every location where scheduled jobs can be defined

On a cPanel server, cron jobs can be defined in four different places. A complete audit requires checking all four. Root access is required — non-root users cannot view other users' crontabs.

1

User crontabs — /var/spool/cron/

Each user with a crontab has a file here named after their username.

bash — as root
# See which users have crontabs
ls /var/spool/cron/
appuser  root

# Read a specific user's crontab (requires root)
crontab -u appuser -l
MAILTO=""
SHELL="/bin/bash"
0 0 * * * /usr/local/bin/php /home/appuser/public_html/backup_services/cron_backup_all.php

# Dump ALL user crontabs at once
cat /var/spool/cron/*
2

System cron drop-in files — /etc/cron.d/

Services and packages install their own cron files here. Each file specifies its own user context.

bash
ls -la /etc/cron.d/
0hourly          backup_jobs_helper  comet_license_sync
cpanel-analytics  cpanel_autossl    csf-cron
lfd-cron          mailman           softaculous
softaculous2

cat /etc/cron.d/*   # read all of them
3

Periodic directories — hourly, daily, weekly, monthly

bash
ls /etc/cron.hourly/   # 0anacron

ls /etc/cron.weekly/   # (empty on this server)
ls /etc/cron.monthly/  # (empty on this server)

for user in $(cut -f1 -d: /etc/passwd); do

Root Cause Identified

clean_user_php_sessions + slow /tmp = PHP worker avalanche

The culprit: 09,39 * * * * /usr/local/cpanel/scripts/clean_user_php_sessions > /dev/null 2>&1 This cPanel maintenance script runs 48 times per day (every 30 minutes). It walks the /tmp filesystem to find and delete expired PHP session files. On this server, /tmp uses a loop device with 131ms latency — 26× slower than normal.

The complete causal chain:

09:39:00

clean_user_php_sessions fires

Cron triggers the script. It begins scanning /tmp for PHP session files matching sess_* pattern.

09:39:01

/tmp filesystem locks during iteration

The loop0 device (131ms per read) creates I/O lock contention. Any other process trying to read/write a session file to /tmp is now blocked.

09:39:02+

PHP workers stall on session I/O

Every active PHP-CGI worker on every site tries to read its session from /tmp on each request (PHP default). All stall and enter S (sleeping) state.

09:39–09:47

Worker pile-up — the avalanche

Apache receives new HTTP requests, spawns new PHP-CGI workers. These stall too. Apache spawns more. Workers accumulate: 5 → 15 → 36 → 43. Load average climbs: 2 → 4 → 7.20.

~09:47

clean_user_php_sessions finishes

Script completes after ~8 minutes. /tmp locks release. All 43 stalled workers drain simultaneously. Load drops from 7.20 to 0.21 within minutes.

09:50

Server fully recovers

Load average: 0.21. Only 10 web connections. vmstat shows si=0 so=0. Server back to normal — until 10:09 when the cycle repeats.

Why psphub was hit hardest: portal-app.example.net had the most active users and therefore the most active PHP session files in /tmp. Workers for that site were the first to stall and accumulated the fastest — 36 out of 43 total PHP workers were for psphub alone.

Live Health Check Commands

The essential toolkit for checking server health in real time

These are the commands used throughout this investigation. Each serves a specific diagnostic purpose. Use these whenever users report slowness.

1

The quick all-in-one health snapshot

Run this first. It tells you load, memory, disk, and PHP worker count in one command.

bash
echo "=== LOAD ===" && uptime && \
echo "=== MEMORY ===" && free -h && \
echo "=== DISK ===" && df -h && \
echo "=== TOP 5 CPU ===" && ps aux --sort=-%cpu | head -6 && \
echo "=== TOP 5 MEM ===" && ps aux --sort=-%mem | head -6 && \
echo "=== PHP WORKERS ===" && ps aux | grep php-cgi | grep -v grep | wc -l
2

Real-time watch — update every 3 seconds

Leave this running in a terminal tab during suspected incidents. Shows load, memory, and PHP worker count live.

bash
watch -n 3 'uptime; echo; free -h; echo; ps aux | grep php-cgi | grep -v grep | wc -l'
# Press Ctrl+C to stop
3

Identify D-state (stuck) processes

Processes in D state are in uninterruptible sleep — waiting on I/O. These directly add to load average even when using no CPU.

bash
# Find all processes in D (uninterruptible sleep) state
ps aux | awk '$8 == "D"' | head -20

# What was found during this incident (11:34 check):
appuser 1765108  D  php-cgi regional-app.example.net   ← stuck on disk
appuser 1765124  D  php-cgi regional-app.example.net   ← stuck on disk

Disk I/O Analysis

How to read iostat and identify storage problems — including the stall pattern discovered at 11:35

iostat -x 1 5 takes 5 samples one second apart. The first sample is a historical average since boot; subsequent samples are real-time intervals.

1

Understanding iostat columns

ColumnMeaningAlert threshold
r/sRead operations per secondContext-dependent
w/sWrite operations per secondContext-dependent
r_awaitAverage wait time for read requests (ms)> 20ms = concern, > 50ms = problem
w_awaitAverage wait time for write requests (ms)> 20ms = concern, > 50ms = problem
%utilPercentage of time device was busy> 80% sustained = saturated
aqu-szAverage queue size (requests waiting)> 1 sustained = queuing
2

First-round iostat results (09:44 — during active investigation)

iostat output — first round
Device    r/s    w/s   r_await  w_await  %util
vda      3.17   7.64   44ms     6.96ms   1.51%
loop0    0.00   0.16   131ms    83.39ms  1.12%  # /tmp — extremely slow

The main disk vda at 44ms is above normal (typical ~5–15ms for VPS SSD) — suggesting VPS storage throttling. But loop0 at 131ms is catastrophic for a device that handles PHP session files on every request.

3

Second-round iostat results (11:35 — new disk stall detected)

iostat live samples — 11:35
# Sample 1 (historical average since boot):
Device   r_await  w_await  %util
vda       50ms      7ms    1.51%
loop0    169ms     93ms    1.12%

# Samples 2-5 (live — 1 second intervals):
avg-cpu: %iowait = 74.75%  74.81%  74.94%  74.56%
vda:     %util  = 100%    100%    100%    100%
vda:     r/s    =   0.00    0.00    0.00    0.00  # ZERO ops but 100% busy!
vda:     w/s    =   0.00    0.00    0.00    0.00
Disk I/O stall pattern: The disk reports 100% utilization but is processing ZERO operations. This is a hung I/O request — one process issued a disk command that the VPS storage backend has not responded to. The entire I/O queue is blocked behind it. Meanwhile vmstat shows b=1 (one process permanently in uninterruptible wait). This is a hosting infrastructure issue.
The contradiction that reveals a true stall: In a healthy system, high %util correlates with high r/s or w/s. When you see %util = 100% alongside r/s = 0, w/s = 0, it means the device is "busy" only in the sense that it's stuck waiting for a response that hasn't come. This is a VPS infrastructure problem, not a software problem.

Fixes & Recommended Actions

Prioritised list — immediate, short-term, and ongoing

Fix 1 — Move PHP sessions off /tmp (Primary fix — eliminates root cause)

The cleanest fix: make PHP store session files on the fast main disk instead of the slow loop0 /tmp device. This means clean_user_php_sessions running on /tmp will no longer affect PHP workers.

bash — WHM approach (recommended)
# Step 1: Create a dedicated session directory for the user
mkdir -p /home/appuser/tmp/sessions
chmod 700 /home/appuser/tmp/sessions
chown appuser:appuser /home/appuser/tmp/sessions

# Step 2: In WHM → MultiPHP INI Editor → User Editor → appuser
# Set:  session.save_path = /home/appuser/tmp/sessions
# Or edit directly:
echo 'session.save_path = "/home/appuser/tmp/sessions"' >> \
  /opt/cpanel/ea-php82/root/etc/php.d/appuser.ini

Fix 2 — Reduce clean_user_php_sessions frequency (Alternative/Additional fix)

bash — edit root crontab
crontab -e   # opens root's crontab in editor

# Find this line:
09,39 * * * * /usr/local/cpanel/scripts/clean_user_php_sessions > /dev/null 2>&1

# Change it to run once nightly at 2 AM instead:
0 2 * * * /usr/local/cpanel/scripts/clean_user_php_sessions > /dev/null 2>&1
Important: cPanel auto-updates (upcp) can overwrite root's crontab. After a cPanel update, verify this change is still in place. Fix 1 (moving session.save_path) is more durable as it survives cPanel updates.

Fix 3 — Reclaim MySQL swap during quiet hours

bash — run at 2–4 AM only
# Disable swap entirely (moves all pages back to RAM), then re-enable
# Only safe if RAM available > current swap used (4.1Gi available, 452Mi in swap = safe)
swapoff -a && swapon -a
# This reclaims MySQL's 206MB from swap, improving query latency

Fix 4 — Block the SSH brute force IP

bash — CSF firewall
# Permanently block the brute force source IP
csf -d 185.208.159.109

# Verify it was added
grep 185.208.159.109 /etc/csf/csf.deny

Fix 5 — Verify the fix worked (monitor at next cron fire)

bash — run at :08 or :38 of any hour
# Start this 1 minute BEFORE the cron fires (:09 or :39)
# Watch load and PHP worker count during clean_user_php_sessions execution
watch -n 3 'uptime; echo; ps aux | grep php-cgi | grep -v grep | wc -l'

# If fix was successful:
#   Load stays < 1.0 during :09/:39 window
#   PHP workers stay < 10
# If still broken:
#   Load spikes to > 4.0 and worker count climbs to 20+

Fix 5 — Investigate psplagos

Both afternoon D-state workers were for regional-app.example.net. Check its slow query log and confirm no long-running database queries are holding connections open.

Fix 6 — Reduce spamd children

spamd is using ~354MB RAM with 3 of 5 slots occupied. If email volume is low, reduce --max-children=2 in WHM to free ~120MB.

Fix 7 — Report stall to host

The afternoon disk stall (100% util, 0 ops/sec) is a VPS infrastructure issue. Contact your host with the iostat output if it recurs frequently.

Key lessons from this incident:

Lesson 1: Timing is evidence

A spike that starts at :09 and :39 every hour — exactly when a cron fires — is not coincidence. Always check the crontab when you see periodic, self-resolving load spikes.

Lesson 2: Logs tell the truth

Zero Apache requests during a load spike immediately eliminates all traffic-based attack theories. Always check the access log before escalating to security response.

Lesson 3: /tmp I/O matters

PHP session files live in /tmp by default. A slow /tmp device creates a hidden performance bottleneck that manifests as load spikes during any cron that accesses /tmp.

Lesson 4: D-state = disk wait

A process in D state is not frozen or dead — it's waiting for the disk. Count D-state processes to quickly identify whether load spikes are I/O-driven rather than CPU-driven.

Lesson 5: 100% util + 0 ops = stall

High disk utilization with zero read/write operations per second is a contradiction that indicates a hung I/O request — not high workload. These require different remedies.

Lesson 6: Audit crons completely

cPanel servers have crons in 4 locations: user crontabs (/var/spool/cron/), /etc/cron.d/, and the periodic directories. A complete investigation must check all of them.
