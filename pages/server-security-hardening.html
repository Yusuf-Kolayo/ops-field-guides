Threat Discovery — Active Brute-Force SSH Attack

Identifying and quantifying an active attack from SSH auth logs

While investigating the server, the SSH authentication log revealed an active brute-force campaign: automated login attempts from external IPs probing the root account and a dictionary of common system usernames.

22

Failed login attempts confirmed in current log file

10

Distinct usernames targeted — all automated wordlist guesses

root

Primary target — attempted 9 times in this window alone

0

Successful unauthorised logins

Step 1 — Count total failed attempts in the log:

bash — run as root on the server
grep -c "Failed password" /var/log/secure
22

Step 2 — The naive username extraction (reveals a parsing quirk):

bash
grep "Failed password" /var/log/secure | awk '{print $9}' | sort | uniq -c | sort -rn | head -10
     13 invalid
      9 root
Why does "invalid" appear as a username? SSH logs use two different line formats:

Known user:  Failed password for root from 1.2.3.4 ...  → field 9 = root
Unknown user: Failed password for invalid user admin from 1.2.3.4 ...  → field 9 = invalid

awk '{print $9}' grabs the word "invalid" instead of the actual username when the account doesn't exist on the server. The real username is in field 11 in that case.

Step 3 — The correct command that handles both log formats:

bash
grep "Failed password" /var/log/secure | awk '{
    if ($9 == "invalid") print $11
    else print $9
}' | sort | uniq -c | sort -rn | head -20
      9 root
      3 orangepi
      3 admin
      1 marketing
      1 green
      1 frappe
      1 elasticsearch
      1 deploy
      1 david
      1 carlos

What this wordlist reveals — these are not random guesses:

Username AttemptedWhy Bots Try ItThreat Type
rootDefault superuser on every Linux serverHigh
adminCommon default on routers, VPS control panels, generic setupsHigh
deployVery common name for deployment service accountsMedium
elasticsearchTargets servers running Elasticsearch (often misconfigured)Medium
frappeTargets servers running ERPNext/Frappe frameworkMedium
orangepiDefault user on Orange Pi single-board computersLow
david, carlos, marketing, greenHuman names / department accounts — sourced from credential leak databasesLow

Step 4 — List the top attacking IP addresses:

bash
grep "Failed password" /var/log/secure | awk '{
    if ($9 == "invalid") print $13
    else print $11
}' | sort | uniq -c | sort -rn | head -10

Step 5 — After completing the investigation, clear shell history to remove sensitive commands:

bash — clear history containing token/credential output
# Clear in-memory history and write empty state to file
history -c && history -w

# Belt-and-suspenders: overwrite the history file directly
cat /dev/null > ~/.bash_history

# Verify: file should be empty (0 lines)
wc -l ~/.bash_history
0 /root/.bash_history

# Verify: in-memory history shows only post-clear commands — this is normal
history
    1  2026-04-13 07:18:37 cat /dev/null > ~/.bash_history
    2  2026-04-13 07:20:26 wc -l ~/.bash_history
    3  2026-04-13 07:20:34 history
# ✓ Only harmless commands remain — the sensitive output is gone
Why the in-memory history still shows 3 entries after clearing: history -c clears in-memory history for the current session. The shell then starts accumulating again immediately. Those 3 entries are only the harmless commands typed after the clear — the sensitive commands are gone. When the session closes, only these harmless entries are written back to the file.
Watch for other open sessions: If multiple SSH sessions are open for the same user simultaneously, each holds its own in-memory history. When those sessions close, they can overwrite the now-empty history file with their own history. Clear history in all active sessions, or run unset HISTFILE in each to prevent them from writing on exit.
Why does this happen? Automated bots continuously scan the entire internet for port 22 (SSH). This isn't targeted at you specifically — it is the background radiation of the internet. Any server on port 22 with a weak or commonly-used root password is eventually compromised. The solution is layered defence: non-standard port + non-root user + fail2ban + SSH keys.
Consequences of a successful brute force: An attacker gaining root SSH access means full server compromise — your databases, application code, .env files with API keys, all hosted client sites, and the ability to use your server as a launch pad to attack other systems. This is a critical risk requiring immediate action.

Changing the SSH Port

Moving SSH off port 22 eliminates 99% of automated attacks instantly

Changing the SSH port from the default 22 to a non-standard port (e.g. 2222) is called security through obscurity. While not a complete defence on its own, it immediately stops all automated bots that only probe port 22.

Critical — do these steps in order. Open your firewall and configure SELinux for the new port before restarting sshd, otherwise you will lock yourself out. Consider keeping a hosting provider web console session open as a backup.
1

Edit the SSH daemon configuration

bash — on the server as root
vi /etc/ssh/sshd_config

# Find the line:
#Port 22

# Change it to (remove the # and update the number):
Port 2222
2

Open the new port in the firewall FIRST

See Section 4 for full iptables details. At minimum, run this before restarting sshd:

bash
iptables -I INPUT -p tcp --dport 2222 -j ACCEPT
3

Tell SELinux about the new port FIRST

See Section 5 for full SELinux details. At minimum, run this before restarting sshd:

bash
semanage port -a -t ssh_port_t -p tcp 2222
4

Restart sshd to apply the new port

bash
systemctl restart sshd

# Confirm sshd is listening on the new port
ss -tlnp | grep sshd
LISTEN  0  128  0.0.0.0:2222  0.0.0.0:*  users:(("sshd",...))
5

Test the new port in a NEW terminal BEFORE closing the current session

bash — run from your Mac, new tab
ssh -p 2222 root@vps01.example.net
# Only close the old terminal AFTER this succeeds
Port choice: Port 2222 is a common alternative and easy to remember. For even less visibility, choose a random high port like 49201 or 51022. Avoid well-known service ports (e.g. 80, 443, 3306, 5432). Ports in the range 1024–65535 are valid choices.
Update your SSH client config: Add a permanent alias in ~/.ssh/config on your Mac so you don't have to remember the port every time:
~/.ssh/config — on your Mac
Host prod-vps
    HostName     vps01.example.net
    User         adminops
    Port         2222
    IdentityFile ~/.ssh/id_ed25519

# Now just type: ssh prod-vps

SSH Lockout Recovery — What Happened and How to Fix It

Getting back into a server when you've locked yourself out

After changing the SSH port and restarting sshd, connection on port 2222 failed completely. The server was inaccessible. Two separate systems were blocking the new port simultaneously.

Root Cause 1 — iptables

The firewall had no rule allowing TCP port 2222. All inbound connections on the new port were dropped silently.

Root Cause 2 — SELinux

SELinux only knew port 22 was legitimate for SSH. It blocked sshd from binding to 2222 with a policy violation.

Recovery Method

Used the hosting provider's web-based VNC/console (accessible from the cPanel/hosting dashboard) — bypasses SSH entirely.

1

Access the server via hosting provider web console

Log in to your hosting control panel (e.g. Ashraaf IT World), navigate to your VPS, and open the VNC Console or Web Terminal. This gives you shell access regardless of firewall or SSH state.

2

Fix iptables to allow port 2222

bash — via web console as root
# Insert rule at position 1 (before any DROP rules)
iptables -I INPUT 1 -p tcp --dport 2222 -j ACCEPT

# Verify the rule was added
iptables -L INPUT -n --line-numbers
3

Fix SELinux to allow port 2222 for SSH

bash
# Register port 2222 as a valid SSH port in SELinux policy
semanage port -a -t ssh_port_t -p tcp 2222

# Confirm it was added
semanage port -l | grep ssh
ssh_port_t   tcp   22, 2222
4

Restart sshd and test from your Mac

bash
systemctl restart sshd

# From your Mac:
ssh -p 2222 root@vps01.example.net
# Should connect successfully now
Always have a backup access method: Before making SSH configuration changes, ensure you have at least one alternative way in — either a web console from your hosting panel, a secondary SSH session that stays open, or a VNC/KVM connection. Never make SSH changes from your only active session without a backup.

iptables Firewall Configuration

This server uses iptables directly — not firewalld

Many CentOS/RHEL guides assume firewalld is active, but on this server it is not running. All firewall rules are managed via iptables directly. Using firewall-cmd will appear to succeed but make no actual difference.

Check which firewall is active first:
bash
systemctl status firewalld
● firewalld.service - firewalld - dynamic firewall daemon
   Loaded: ...
   Active: inactive (dead)   # ← on this server

systemctl status iptables
● iptables.service - IPv4 firewall with iptables
   Active: active (exited)   # ← this one is running
1

View current iptables rules (with line numbers)

bash
iptables -L INPUT -n --line-numbers
2

Allow the new SSH port (insert at top to avoid DROP rules)

bash
# -I inserts at the top (before any DROP/REJECT rules)
iptables -I INPUT -p tcp --dport 2222 -j ACCEPT
3

Save the rules so they survive a reboot

bash
iptables-save > /etc/sysconfig/iptables

# Verify the rule was saved
grep 2222 /etc/sysconfig/iptables
-A INPUT -p tcp -m tcp --dport 2222 -j ACCEPT
Difference between -I and -A: -I inserts a rule at the top of the chain (evaluated first). -A appends to the bottom (evaluated last). If your chain ends with a DROP policy, -A may never be reached. Always use -I when adding ACCEPT rules to ensure they are processed before any DROP rules.
CommandPurpose
iptables -L INPUT -n --line-numbersView current INPUT chain rules with line numbers
iptables -I INPUT -p tcp --dport PORT -j ACCEPTAllow incoming TCP on a port (insert at top)
iptables -D INPUT LINE_NUMBERDelete a rule by its line number
iptables-save > /etc/sysconfig/iptablesPersist rules across reboots (CentOS/RHEL)
iptables -F INPUTFlush all INPUT rules (dangerous — use with caution)

SELinux Configuration for Non-Standard SSH Port

Why SELinux blocks sshd on non-default ports and how to fix it

SELinux (Security Enhanced Linux) enforces mandatory access controls on this server. By default, it only permits sshd to bind to port 22. Attempting to use any other port causes sshd to fail silently or with a policy violation error — even if the firewall allows the port.

How to tell if SELinux is the problem: If sshd fails to start or refuses to listen on the new port even after setting Port 2222 in sshd_config, SELinux is almost certainly blocking it. Check: ausearch -c sshd --raw | audit2why or grep sshd /var/log/audit/audit.log | grep denied
1

Check SELinux status

bash
getenforce
Enforcing   # SELinux is active and blocking policy violations
2

See which ports SELinux currently allows for SSH

bash
semanage port -l | grep ssh
ssh_port_t   tcp   22   # only port 22 is allowed by default
3

Add port 2222 to SELinux's allowed SSH ports

bash
semanage port -a -t ssh_port_t -p tcp 2222

# Verify it was added
semanage port -l | grep ssh
ssh_port_t   tcp   22, 2222   # ✓ both ports now allowed
Do NOT disable SELinux: It is a critical security layer. The correct response is never setenforce 0 for production. That disables all SELinux protections server-wide. Always use semanage to grant specific permissions instead.
CommandPurpose
getenforceShow current SELinux mode (Enforcing / Permissive / Disabled)
semanage port -l | grep sshList all ports allowed for SSH in SELinux policy
semanage port -a -t ssh_port_t -p tcp PORTAdd a new allowed SSH port
semanage port -d -t ssh_port_t -p tcp PORTRemove an allowed SSH port
ausearch -c sshd --raw | audit2whyDiagnose SELinux denials related to sshd

Creating a Non-Root SSH User

Never SSH directly as root — use a regular user with sudo access

Direct root SSH access is a significant risk. If any attacker guesses the password, they have immediate, unlimited control. The best practice is to create a named user account, grant it sudo privileges, and disable direct root login over SSH.

1

Create the new user

bash — as root
adduser adminuser

# On CentOS/RHEL, adduser does NOT set a password automatically
# You must run passwd separately:
passwd adminuser
New password:
Retype new password:
passwd: all authentication tokens updated successfully.
2

Grant sudo access by adding to the wheel group

On CentOS/RHEL, the sudo group is called wheel (not sudo as on Ubuntu/Debian).

bash
usermod -aG wheel adminuser

# Verify wheel group membership
groups adminuser
adminuser : adminuser wheel
3

Test the new user in a separate terminal

bash — new terminal on your Mac
# SSH in as adminuser
ssh -p 2222 adminuser@vps01.example.net

# Once connected, escalate to root
sudo su -
[sudo] password for adminuser:
[root@vps01 ~]#   # ✓ full root access achieved
Why sudo su - and not just sudo command? For most day-to-day admin tasks, using sudo command is preferred as it gives a minimal footprint (only elevated for one command). However, sudo su - gives you a full root shell when you need to run many root commands in sequence. The - ensures root's full environment is loaded. Both are fine — use what fits the task.

Disabling Direct Root SSH Login

The single most impactful SSH security setting

Once you have a working non-root user (Section 6), disable root login over SSH. This makes brute-force attacks against root impossible — even if someone guesses the password, the login will be rejected outright.

Before disabling root login: Confirm in a separate terminal that you can ssh -p 2222 adminuser@vps01.example.net and then sudo su - successfully. Only proceed when this is verified and working. Otherwise you will permanently lock yourself out of the server.
1

Edit sshd_config to set PermitRootLogin to no

bash — as root
# One-liner that replaces any existing PermitRootLogin setting
sed -i 's/.*PermitRootLogin.*/PermitRootLogin no/' /etc/ssh/sshd_config

# Verify the change
grep PermitRootLogin /etc/ssh/sshd_config
PermitRootLogin no
2

Restart sshd

bash
systemctl restart sshd
3

Test that root login is now rejected

bash — from your Mac
ssh -p 2222 root@vps01.example.net
root@vps01.example.net: Permission denied (publickey,gssapi-keyex,gssapi-with-mic).
# ✓ root login rejected — expected and correct

# Your normal workflow now:
ssh -p 2222 adminuser@vps01.example.net
sudo su -   # ← escalate when root access needed
PermitRootLogin ValueBehaviourRecommendation
yes Root can login with password or key Never use
prohibit-password Root can login with SSH key only (no password) Acceptable
no Root SSH login completely disabled Recommended

Renaming an Active SSH User

Why usermod fails when the user is logged in — and how to work around it

For additional obscurity, it is useful to rename the SSH user to something less common than a first name. However, renaming a user who is currently logged in is blocked by the system — Linux won't let you modify files for a user with active processes.

The error you will see if you try wrong: 
usermod -l adminops adminuser
usermod: user adminuser is currently used by process 1253303
This happens when you are SSH'd in as adminuser and try to rename adminuser in the same session, even via sudo su -. The original SSH session is still tied to the adminuser user.
1

Open a separate terminal and SSH directly as root

This gives you a root session that is NOT associated with the adminuser user process.

bash — NEW terminal tab on Mac
# Temporarily re-enable root login if PermitRootLogin is no:
# (from the active adminuser session via sudo)
sudo sed -i 's/PermitRootLogin no/PermitRootLogin yes/' /etc/ssh/sshd_config
sudo systemctl restart sshd

# Then from a NEW terminal:
ssh -p 2222 root@vps01.example.net
2

Rename the username and home directory

bash — root session (NOT via adminuser sudo)
# Step 1: rename the login name
usermod -l adminops adminuser

# Step 2: move and rename the home directory
usermod -d /home/adminops -m adminops

# Step 3: rename the primary group to match
groupmod -n adminops adminuser

# Step 4: verify everything looks correct
id adminops
uid=1002(adminops) gid=1002(adminops) groups=1002(adminops),10(wheel)
3

Test the renamed user from a new Mac terminal

bash — Mac terminal
ssh -p 2222 adminops@vps01.example.net
[adminops@vps01 ~]$   # ✓ login successful with new name

sudo su -
[root@vps01 ~]#   # ✓ sudo still works
4

Re-disable root SSH login

bash — from root or via sudo
sed -i 's/.*PermitRootLogin.*/PermitRootLogin no/' /etc/ssh/sshd_config
systemctl restart sshd
Alternative: rename from the hosting web console — If re-enabling root login feels risky, use the hosting provider's web console (VNC/browser terminal) instead. This gives you a root terminal that is completely independent of any SSH user session.

Installing and Configuring fail2ban

Automatically block IPs after repeated failed login attempts

fail2ban monitors auth logs and uses iptables to automatically ban IP addresses that show brute-force behaviour. After 5 failed attempts in 10 minutes, the offending IP is blocked for 24 hours. It is your automated first responder against SSH attacks.

1

Install fail2ban via EPEL

fail2ban is not in the default CentOS repos — it lives in the EPEL (Extra Packages for Enterprise Linux) repository.

bash — as root
# Step 1: enable the EPEL repository
yum install -y epel-release

# Step 2: install fail2ban
yum install -y fail2ban

# Verify installation
fail2ban-client --version
Fail2Ban v0.11.2
2

Create a local jail configuration for SSH

Always create a jail.local file rather than editing jail.conf directly — local overrides survive package updates.

bash
cat > /etc/fail2ban/jail.local << 'EOF'
[DEFAULT]
# Ban IPs for 24 hours (86400 seconds)
bantime  = 86400
# Look back 10 minutes for failures
findtime = 600
# Allow 5 failures before banning
maxretry = 5
# Use iptables (matches this server's firewall)
banaction = iptables-multiport

[sshd]
enabled  = true
port     = 2222
logpath  = /var/log/secure
maxretry = 5
EOF
3

Enable and start fail2ban

bash
# Enable on boot and start now
systemctl enable fail2ban
systemctl start fail2ban

# Verify it is protecting SSH
fail2ban-client status sshd
Status for the jail: sshd
|- Filter
|  |- Currently failed: 2
|  |- Total failed:     740
|  `- Journal matches:  _SYSTEMD_UNIT=sshd.service + _COMM=sshd
`- Actions
   |- Currently banned: 14
   |- Total banned:     28
   `- Banned IP list: 185.234.218.97 45.142.212.100 ...

Essential fail2ban management commands:

CommandPurpose
fail2ban-client status sshdSee current bans and stats for SSH jail
fail2ban-client statusOverview of all active jails
fail2ban-client set sshd unbanip 1.2.3.4Unban a specific IP (e.g., your own by mistake)
fail2ban-client set sshd banip 5.6.7.8Manually ban an IP immediately
tail -f /var/log/fail2ban.logWatch fail2ban activity in real time
fail2ban-client reloadReload config after making changes to jail.local
Watch out — you can ban yourself: If you mistype your password 5 times from your Mac, your home IP will be banned for 24 hours. Unban yourself with: fail2ban-client set sshd unbanip YOUR_IP. To find your IP from the Mac: curl -4 ifconfig.me. Consider whitelisting your home IP in jail.local using ignoreip = 127.0.0.1/8 YOUR_HOME_IP.

SSH Key Authentication

The strongest form of SSH authentication — eliminates password brute force entirely

With SSH key authentication, a cryptographic key pair replaces passwords entirely. Even if someone knows your username and port, they cannot log in without possessing your private key file. Combined with PasswordAuthentication no, brute-force attacks become mathematically impossible.

1

Generate an SSH key pair on your Mac

bash — on your Mac
# Generate an Ed25519 key (modern, fast, secure)
ssh-keygen -t ed25519 -C "mac@psphub-server"

Generating public/private ed25519 key pair.
Enter file in which to save the key (/Users/mac/.ssh/id_ed25519):
Enter passphrase (empty for no passphrase): [enter a passphrase for extra protection]
Your identification has been saved in /Users/mac/.ssh/id_ed25519
Your public key has been saved in /Users/mac/.ssh/id_ed25519.pub
2

Copy your public key to the server

bash — on your Mac
ssh-copy-id -p 2222 adminops@vps01.example.net

# Or manually if ssh-copy-id isn't available:
cat ~/.ssh/id_ed25519.pub | ssh -p 2222 adminops@vps01.example.net \
  "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"
3

Test passwordless login

bash — on your Mac
ssh -p 2222 adminops@vps01.example.net
# Should connect without asking for a password
[adminops@vps01 ~]$
4

Disable password authentication entirely (optional hardening)

Only do this after confirming SSH key login works perfectly.

bash — on the server as root
sed -i 's/.*PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config
systemctl restart sshd

# Now ONLY SSH key holders can log in — brute force is impossible
Store your private key safely: If you lose ~/.ssh/id_ed25519 (your private key) and password authentication is disabled, you will be permanently locked out. Keep a backup of the key in a secure, encrypted location (e.g. your password manager or an encrypted USB drive). Also keep your hosting provider's web console access working as a recovery option.

Rotating Exposed Secrets

Tokens and secrets visible in shell history or terminal output must be rotated immediately

During the session, a CRON_SECRET_TOKEN was visible in terminal output and shell history. Any secret that appears in a terminal (even a private one) should be treated as potentially compromised and rotated. Shell history can be captured by terminal recording tools, screenshots, and session logs.

The exposed token: The CRON_SECRET_TOKEN value was visible in command-line output. If this token is discovered, an attacker could call the cron reconciliation endpoint arbitrarily — triggering unintended financial data processing. Rotate it immediately.
1

Generate a new cryptographically random token

bash — on the server as root
openssl rand -hex 32
a9f3b2c1e4d7a6b5c8d1e3f2a4b7c0d9e2f1a3b4c5d6e7f8a9b0c1d2e3f4a5
# Copy this value — it will be your new token
2

Update the .env file in the Laravel application

bash — navigate to the Laravel app directory
cd /path/to/your/laravel/app

# Edit the .env file directly
vi .env

# Find and replace the line:
CRON_SECRET_TOKEN=72cd44dd7b8c1a2dbbfe5cba9a3c084d

# Change it to your new token:
CRON_SECRET_TOKEN=a9f3b2c1e4d7a6b5c8d1e3f2a4b7c0d9e2f1a3b4c5d6e7f8a9b0c1d2e3f4a5
3

Clear Laravel's config cache so it picks up the new value

bash
php artisan config:clear
Configuration cache cleared successfully.

# Optionally rebuild the cache for better performance
php artisan config:cache
4

Update any cron jobs or scripts that use the old token

Search for the old token in any automation scripts, scheduled job definitions, or monitoring tools and replace it with the new value.

bash — search for old token usage
# Check crontab entries
crontab -l

# Search all project files (from Mac) for the old token string
grep -r "72cd44dd" /path/to/project --include "*.sh" --include "*.env*"
5

Clear shell history containing the exposed token

bash — on the server
# Clear bash history for the current user
history -c && history -w

# Also clear root's history file if needed
cat /dev/null > ~/.bash_history
General secret hygiene rules:
  • Never pass secrets as command-line arguments if avoidable — they appear in process listings (ps aux)
  • Never commit .env files to Git — add .env to .gitignore
  • Use environment variables or a secrets manager (e.g. AWS Secrets Manager, Laravel Vault) for production
  • Rotate all secrets after any suspected exposure — it takes minutes and eliminates the risk
  • Treat any secret visible in a shared terminal session as compromised

Security impact of each layer — from least to most impactful:

Security MeasureStopsDifficultyImpact
Non-standard SSH port All automated bots scanning port 22 Easy High (reduces noise by ~99%)
PermitRootLogin no Direct root brute force — most targeted account Easy High
fail2ban Persistent attackers who discover the new port Moderate High
SSH key authentication Password brute force entirely — even with unlimited attempts Moderate Maximum

Defence-in-depth principle: No single measure is enough. The goal is to make each layer independent — if one fails, the others still protect you. Port change kills automated scanning. Disabling root login removes the most targeted account. fail2ban blocks persistent attackers. SSH keys make brute force mathematically infeasible. Together, these four measures reduce your attack surface to near zero.
