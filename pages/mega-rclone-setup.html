Overview

Why rclone + MEGA for database backups

MEGA is a zero-knowledge cloud storage provider offering 20GB free to anyone. Unlike GitHub's file size limits or complex REST APIs, MEGA provides end-to-end encryption, large file support, and unlimited bandwidth. rclone is a command-line tool that abstracts the MEGA API, making it simple to upload, sync, and manage files from any server.

Why This Approach Works

End-to-End Encryption

Your backups are encrypted on your machine before transmission. MEGA cannot decrypt them.

Large File Support

Upload SQL dumps up to 400MB+ on the free plan. No GitHub file size restrictions.

Simple CLI Integration

rclone provides a straightforward command-line interface for integration into PHP and cron jobs.

Battle-Tested

rclone is actively maintained, widely adopted, and proven reliable for production backups.

MEGA's Business Model: MEGA makes money from premium accounts ($5–20/month). The 20GB free tier is sustainable long-term. Your backups are yours alone — MEGA's "zero-knowledge" architecture means even MEGA employees cannot access them.

Installing rclone

Deploy rclone on your VPS

rclone provides an official installation script that auto-detects your OS and installs the latest version.

1

SSH into Your VPS

Connect as root or a sudo-capable user.

bash 
ssh -p 2222 root@your-vps-ip
2

Run the Official Installation Script

This script downloads, verifies, and installs rclone automatically:

bash 
curl https://rclone.org/install.sh | sudo bash

This will download the latest rclone binary (currently v1.74.3) and install it to /usr/bin/rclone.

3

Verify Installation

Check that rclone is installed and accessible:

bash 
rclone version
rclone v1.74.3
- os/version: almalinux 8.10 (64 bit)
- os/kernel: 4.18.0-553.el8_10.x86_64 (x86_64)
- go/version: go1.26.4

✓ Installation successful! You now have rclone ready for configuration.

Installation Location: rclone is installed to /usr/bin/rclone. When calling rclone from PHP via shell_exec(), always use the full path: /usr/bin/rclone rather than just rclone. This ensures the command is found regardless of the PHP process's PATH environment variable.

MEGA Authentication

Configure rclone to access your MEGA account

rclone stores MEGA credentials locally in an encrypted configuration file. The interactive setup guides you through account linking.

1

Launch rclone Config Wizard

Start the interactive configuration process:

bash 
rclone config
2

Create a New Remote

When prompted, type n to create a new remote:

bash 
No remotes found, make a new one?
n) New remote
s) Set configuration password
q) Quit config
n/s/q> n
3

Name Your Remote

Name the remote mega (this is what you'll reference in commands):

bash 
Enter name for new remote.
name> mega
4

Select MEGA Storage Type

From the storage type list, select 39 for MEGA:

bash 
...
39 / Mega
   \ (mega)
...
Storage> 39
5

Enter Your MEGA Credentials

Input your MEGA account email and password:

bash 
Option user.
User name.
Enter a value.
user> your-mega-email@example.com

Option pass.
Password.
Choose an alternative below.
y) Yes, type in my own password
g) Generate random password
y/g> y

Enter the password:
password: (type your MEGA password — not visible)

Clarification: if rclone asks for the password twice, it is usually asking you to confirm your existing MEGA account password because the input is hidden. It is not creating a new MEGA password or a separate MEGA account.

6

Optional: 2FA Configuration

If your MEGA account has 2-factor authentication, enter your 2FA code. Otherwise, press Enter to skip:

bash

Skip Advanced Settings

Type n to skip advanced configuration (defaults work fine):

bash

y/n> n

Confirm and Save

y/e/d> y
9

Type q to quit the config wizard:

Where is the config stored? rclone saves your encrypted credentials to ~/.config/rclone/rclone.conf. For root user, this is /root/.config/rclone/rclone.conf. If you run rclone from a cron job as root, this file must be accessible in the root home directory.
MEGA Account Considerations: We recommend using a dedicated MEGA account for backups (separate from personal data). This keeps your backup storage isolated and makes it easier to audit access. If your personal MEGA account is compromised, your backups remain protected.

Verify Connection

Test that rclone can reach MEGA

Before proceeding with backups, verify your MEGA connection works and see what's currently in your account.

1

List Your MEGA Files

See all files and folders in your MEGA account:

bash 
rclone ls mega:
# If your account is empty, output will be blank (normal for first-time setup)
2

List Directories Only

For a faster check, list only directories (not all files):

bash 
rclone lsd mega:
# Output: (empty for new account)

✓ If this command succeeds with no output, your MEGA connection is working! Your account is empty, which is expected for a fresh setup.

3

Connection Failed? Debug with Verbose Output

If the above commands hang or return errors, get detailed debug information:

bash 
rclone -vv ls mega: 2>&1 | head -50

The -vv flag enables verbose logging. Look for lines containing "ERROR" or "timeout" to diagnose connection issues.

First Connection May Be Slow: On the first connection to MEGA, rclone performs an initial authentication handshake. This can take 10–30 seconds. The command may appear to hang. Be patient — this is normal. Subsequent connections are much faster as rclone caches the session.

Optional: Nested Organization: For large deployments with many databases, you can create sub-folders: rclone mkdir mega:backups/production, rclone mkdir mega:backups/staging, etc. In this guide, we use a flat mega:backups/ structure for simplicity.

Test Uploads

Verify file uploads work correctly

Before running the production backup script, test that rclone can upload files to MEGA successfully.

1

Create a Test File

Create a small test file locally:

bash 
echo "test backup content" > test.sql.gz
2

Upload to MEGA

Copy the test file to the backups folder:

bash 
rclone copy test.sql.gz mega:backups/test.sql.gz

This command uploads test.sql.gz to mega:backups/.

3

Verify Upload

List files in the MEGA backups folder to confirm the upload:

bash 
rclone ls mega:backups/
       19 test.sql.gz

✓ Upload successful! Your 19-byte test file is now in MEGA.

4

Clean Up

Delete the test files (both local and remote):

bash 
rm test.sql.gz
rclone delete mega:backups/test.sql.gz
Speed Estimation: A 100MB SQL dump typically uploads in 30–60 seconds depending on your network speed. MEGA supports parallel uploads, so rclone can be faster with multiple files. On slow VPS connections (1Mbps upload), expect 1–5 minutes per 100MB file.

PHP Integration

Call rclone from your PHP backup script

The PHP backup script uses shell_exec() to invoke rclone commands. Here's how to integrate rclone into your database backup workflow.

PHP Configuration Constants

php 
// At the top of your backup script (e.g., cron_backup_all.php)
const RCLONE_REMOTE = 'mega';          // Name of your rclone remote
const RCLONE_BACKUP_DIR = 'backups';    // Folder in MEGA
define('SQL_DIR', __DIR__ . '/sql_files'); // Absolute path for temporary folder

Core Upload Function

php 
// Inside your backup loop, after compressing the SQL file:

// Build the rclone remote path
$remotePath = RCLONE_REMOTE . ':' . RCLONE_BACKUP_DIR . '/' . basename($gzFilename);

// Explicitly set the config file path (CRITICAL for web/cron execution)
$rcloneConfig = __DIR__ . '/rclone.conf';

// Use full path to rclone and pass the config flag
$command = "/usr/bin/rclone --config " . escapeshellarg($rcloneConfig) . " copy " . 
            escapeshellarg($gzFilename) . " " . escapeshellarg($remotePath) . " 2>&1";

// Execute the command and capture output and exit code
exec($command, $outputArray, $exitCode);
$output = implode("\n", $outputArray);

// Check for errors using the exit code (0 means success)
if ($exitCode === 0) {
    echo "Backup for $name uploaded to MEGA successfully.\n";
} else {
    echo "Error uploading $name (Exit Code $exitCode): " . $output . "\n";
}
Critical: Use Full Path to rclone
Always use /usr/bin/rclone in your PHP code, not just rclone. When PHP's shell_exec() runs as the www-data or apache user, the PATH environment may not include where rclone is installed. Using the full path ensures the command is always found.
escapeshellarg() is Essential: The escapeshellarg() function prevents shell injection attacks. Always wrap user-controlled paths and filenames with it before passing to shell_exec().
Why capture stderr (2>&1)? The 2>&1 at the end of the command redirects stderr to stdout. This ensures error messages are captured in $output, making debugging easier.

File Overwrite Strategy

Keep storage costs minimal with smart overwrites

On a 20GB free MEGA plan, you must manage storage carefully. By default, rclone copy skips files that already exist. To keep backups fresh and storage optimal, use the --force flag to overwrite old backups with new ones.

Two Strategies

Strategy Command Storage Usage Use Case
Always Overwrite (Recommended) rclone copy --force Stays constant at 1 backup per DB Limited free tier (20GB)
Keep All Versions rclone copy (without --force) Grows indefinitely When you need backup history
Versioned (Date-Based) Rename files with timestamp Grows by 1 backup per day per DB Balance: keep recent history

Implementation: The --force Flag

Modify your PHP upload command to include --force:

php 
$command = "/usr/bin/rclone --config " . escapeshellarg($rcloneConfig) . " copy --force " . escapeshellarg($gzFilename) . 
            " " . escapeshellarg($remotePath) . " 2>&1";

Behavior Comparison

Without --force (default):

  • Day 1: Upload century_backups_.sql.gz (100MB) → MEGA has 100MB
  • Day 2: Try to upload century_backups_.sql.gz → File exists, skipped. MEGA still has 100MB (old version)
  • Day 3+: Continue skipping. Old backup is stale!

With --force:

  • Day 1: Upload century_backups_.sql.gz (100MB) → MEGA has 100MB
  • Day 2: Upload century_backups_.sql.gz (105MB, newer) → MEGA overwrites, now has 105MB (latest version)
  • Day 3+: Always have the most recent backup, storage stays ~constant
Recommended Setting: Use --force for free tier users. Your 20GB storage always holds the current backups, and you're never limited by accumulated old versions.

Cron Automation

Schedule backups to run automatically

Once your backup script is tested and working, set up a cron job to run it automatically on a schedule (e.g., daily at 2 AM).

1

Open the Crontab Editor

Edit the cron schedule for the root user (who can access all databases):

bash 
crontab -e

This opens your default text editor (usually vi or nano). If you see an error like "no crontab," that's fine — cron will create one.

2

Add the Cron Job

Insert this line at the end of the file (adjust path to your script):

bash 
# Run database backups daily at 2:00 AM
0 2 * * * cd /home/appuser/public_html/backup_services && /usr/bin/php cron_backup_all.php >> backup.log 2>&1

Breaking this down:

  • 0 2 * * * = Every day at 02:00 (2 AM) Africa/Lagos timezone
  • cd .../backup_services = Navigate to the script directory
  • /usr/bin/php cron_backup_all.php = Run the backup script with full PHP path
  • >> backup.log 2>&1 = Append all output (stdout and stderr) to backup.log
3

Save and Exit

Save the file:

  • In nano: Press Ctrl+O, press Enter, then Ctrl+X
  • In vi/vim: Type :wq and press Enter
4

Verify the Cron Job

List your active cron jobs to confirm it was added:

bash 
crontab -l
0 2 * * * cd /home/appuser/public_html/backup_services && /usr/bin/php cron_backup_all.php >> backup.log 2>&1

✓ Cron job successfully scheduled! Your backups will now run automatically daily.

Common Cron Schedules

Cron Expression Frequency Use Case
0 2 * * * Every day at 2 AM Default — low server impact, minimal storage growth
0 */6 * * * Every 6 hours (2 AM, 8 AM, 2 PM, 8 PM) Frequent backups for critical data
0 0 * * 0 Every Sunday at midnight Weekly backups (less frequent, more storage in MEGA)
30 1 * * * Every day at 1:30 AM Offset from "2 AM" for load distribution
Cron Timezone: By default, cron uses the server's system timezone. Your backups are configured to run at Africa/Lagos time. To verify the server timezone, run: timedatectl show --property=Timezone --value
Output Logging: The >> backup.log 2>&1 redirect is critical. Without it, cron emails you output on every run, which can spam your mailbox. Always redirect cron output to a log file.

Troubleshooting

Diagnose and fix common issues

Problem Cause Solution
"rclone: command not found" rclone is not in the system PATH or not installed Use full path: /usr/bin/rclone. Verify: which rclone
Script says "Success" in Web/Cron but files aren't in MEGA Web/Cron user (e.g., www-data) cannot read root's rclone.conf and fails silently. Copy /root/.config/rclone/rclone.conf to your project folder, run chmod 644 rclone.conf, and use the --config flag in PHP (see Section 8).
"Permission denied" on SQL dump in Web/Cron Script is executing in a different working directory (e.g., /) and relative paths fail. Always use absolute paths with __DIR__ (e.g., __DIR__ . '/sql_files') in your PHP script.
"Permission denied" on SQL dump MySQL user cannot write to sql_files/ directory Ensure directory exists with correct permissions: mkdir -p sql_files && chmod 755 sql_files
Upload hangs or times out Slow network or very large file (100MB+) Check network: ping mega.nz. Test with smaller file first. Increase PHP timeout: set_time_limit(0)
"Access Denied" when listing MEGA MEGA credentials are wrong or account is locked Reconfigure: rclone config and re-enter email/password. Check for MEGA account warnings
Cron job not running Cron syntax error or service not enabled Verify syntax: crontab -l. Check cron is running: systemctl status crond

Debug rclone Connection

bash 
# Enable verbose logging for rclone
rclone -vv ls mega: 2>&1 | head -100

# Look for lines containing "ERROR" or "timeout"
